### BUG-MED-08 - Release quality gate treated placeholder Google Vision config as configured

Status: Mitigated.

- Evidence:
  - API runtime uses `isConfigured(...)`, which treats `YOUR_VALUE_HERE` as missing.
  - Earlier release quality script only checked truthiness, so placeholder values could look configured.
  - `scripts/release-quality-check.mjs` now uses the same placeholder-aware check.
  - Latest `npm run quality:release` passed with warning: Google Vision credentials are missing and OCR/tagging may use fallback behavior.
- Impact:
  - Demo team may expect live OCR/category tags while runtime falls back to empty tags.
  - Judge could catch mismatch between slide wording and actual configured behavior.
- Recommended fix:
  - Before defense, either set a real `GOOGLE_VISION_API_KEY` or demo matching without relying on live OCR.
  - Keep wording as Google Vision assisted OCR when configured, not guaranteed live OCR in every environment.

### BUG-MED-09 - Release quality gate could misread env precedence and whitespace placeholders

Status: Fixed.

- Evidence:
  - `scripts/release-quality-check.mjs` loaded root `.env` before `apps/api-node/.env` and did not allow the API env file to replace placeholder values.
  - `isConfigured(...)` compared raw values to `YOUR_VALUE_HERE`, so whitespace-padded placeholders could look configured.
- Impact:
  - A developer with root `.env` placeholders and real API credentials in `apps/api-node/.env` could get false Cloudinary/Google Vision warnings.
  - Conversely, whitespace-padded placeholders could make runtime/release checks look configured when they are not.
- Recommended fix:
  - Done: API `.env` can override unconfigured placeholder values from root `.env` in release quality checks.
  - Done: both runtime `isConfigured` and release quality `isConfigured` trim before placeholder comparison.
  - Done: text scan now includes `scripts` while skipping the quality script itself to avoid self-reporting its known-bad pattern list.

### BUG-MED-10 - API CORS was too permissive for production-like demos

Status: Fixed.

- Evidence:
  - Previous `apps/api-node/src/app.ts` used plain `cors()`, which accepts any browser origin.
  - `apps/api-node/src/app.ts` now restricts CORS origins to `FRONTEND_URL` and comma-separated `SOCKET_CORS_ORIGIN`.
  - Localhost, `127.0.0.1`, and Android emulator `10.0.2.2` are allowed only outside production.
- Impact:
  - Open CORS weakens the security story during defense and can be questioned by judges even when bearer tokens are still required.
  - A production/campus pilot should avoid allowing arbitrary browser origins by default.
- Recommended fix:
  - Done for Node API.
  - Keep `FRONTEND_URL` and `SOCKET_CORS_ORIGIN` accurate in demo/deploy env.
  - Socket.IO already uses `SOCKET_CORS_ORIGIN`; review multi-origin production deployment if needed.
